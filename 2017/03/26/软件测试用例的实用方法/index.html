软件测试用例的实用方法

发表于 2017-03-26 | 分类于 软件测试

最近在读微软的软件测试之道，看到测试用例这一章,这一章主要讲的的是测试设计的基本原理，做个总结吧，biubiubiu~

实践良好的软件设计和测试设计

测试设计和好的软件设计有很多相似之处，测试设计需要从计划和问题解决等方面决定做哪些测试，以及哪种测试在验证功能和确认错误路径处理得当上最有效。测试设计中最重要的方面之一就是能预见用户的需要和期望，然后创建能够恰当地处理这些需要的测试。

使用测试模式

  • 测试模式解决了测试工程师在设计测试上的很多普通问题并提供了有针对性的方针和策略，有一些测试是结构测试的方法，有一些是启发性的方法，还有一些是各种想法的符合或完全不同类的东西。测试模式的重要性在于测试工作人员可以用它来交流一个测试方法的内容，而且是用一种可以被理解和执行的方法来共享不同的测试设计技术。
  • 测试模式共享的常用形式是模板，微软常使用的一种简易模板包括以下属性：名称、问题、分析、设计、预言、用例、缺陷和局限、相关的模式

估计测试时间

测试时间估计的因素：历史数据，复杂度，商业目标，一致性和服从性

从测试开始

测试设计的一个起点可以是对软件的功能规格和需求的审查。如果已经具备了良好的产品功能需求，基于需求的测试会是一个不错的出发点，如果没有需求，那么最好的测试出发点就是提出问题。

搜出问题

如果代码已经存在，但是需求或者功能没有到位，最好的开始测试方法就是运行这个程序。探索式测试，也就是测试和设计测试同时进行，可以很大程度的影响测试设计，而且对于整个测试设计过程也是很有利的一部分。

制定测试策略

  • 一个测试策略引导着测试设计，而且可以为测试团队的测试设计提供方向。这个策略包括测试的种类、进程，和测试时测试团队采用什么方法。它包括对风险的评估，以帮助团队决定最有可能发生在哪里或者某些组件是否可能需要更加全面彻底的测试。
  • 测试策略实例的属性：介绍、需求规格、关键情景、测试方法、可交付使用的测试、培训。

考虑可测试性

可测试性是指软件可以完全被有效测试的程度，采用不同的设计，选择简单的算法，使用测试“钩子”和让内部变量可见都是提高可测试性的例子。

同时用好数据和坏数据进行测试

测试用例一般包括验证测试（使用期望的输入来验证产品功能的测试）和错误避免测试（是用预期之外的数据来检验产品是否能适当处理的测试）。验证测试是必要的，它证明了应用可以按照设计原意工作；而错误避免测试可能会更重要一些，应用程序需要健全并能够在自身不出错的情况下处理坏的数据。

测试用例设计中应考虑的其他因素

设计测试用例时，需要考虑产品的范畴、用户基数的大小、测试团队的大小以及测试团队的技能。

黑盒测试、白盒测试和灰盒测试

  • 黑盒测试是指一种设计测试用例的方法，它不需要知道任何关于程序内部是如何实现各种功能的知识，黑盒测试的方法是模拟和预估用户会如何使用这个产品的有效方法。但是单纯的黑盒测试方法也经常会导致过度测试一个应用程序的某些部分而对另一些部分的测试不足够。
  • 白盒测试则是通过对应用程序内部代码或者用户看不到的模式进行分析，并以此设计测试用例，一般凭借白盒测试设计的测试用例都非常详尽，但总是会错过关键的终端用户使用场景。
  • 灰盒测试首先是从用户关心的角度使用黑盒测试，然后再利用白盒测试方法保证测试用例能够有效并全面的覆盖被测试对象。
